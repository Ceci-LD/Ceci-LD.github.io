Box jump AI using NEAT

  • Project Status: Complete
  • Project Type: Personal
  • Project Duration: ~4 days
  • Software Used: VS Code
  • Languages / Libraries Used: JavaScript, p5.js, p5.collide2d.js

My first journey into machine learning did not begin with this project. I took a machine learning course during the 6th semester of my Bachelor studies, learning much of the basics and theory for creating neural networks, and using algorithms for the different types of machine "learning". I decided to venture off deeper into the subject and create my own AI that can tackle problems. Initially I started recreating Flappy Bird using the NEAT algorithm, but encountered countless of obstacles until the whole project was rendered useless and unable to work in any way. I messed up big time, but came out with a better understanding of the subject.

Box jump is a simple 2D game where the player/AI has to jump over incoming boxes. The boxes have a random chance to spawn in one of four different positions. After the initial stage was complete and the player could move and avoid obstacles, it was time to implement the AI.

The first task was to create the input layer from the box to the neural network. There are total of 7 inputs: x and y position of the player box and the closest enemy box, distance between the closest enemy box and the player box. The hardest part was creating the population and visualizing each individual box. After the connections were done I implemented a small function to change between training the boxes and showing the best box so far.

Flocking Algorithm

  • Project Status: Complete
  • Project Type: Personal
  • Project Duration: ~1 days
  • Software Used: VS Code
  • Languages / Libraries Used: JavaScript, p5.js

After seeing the 1986 flocking simulation, I decided to read Craig Reynolds' (the inventor) background regarding the Boids. Thereafter I wanted to recreate it by just following the three rules: Alignment, Cohesion and Separation.

I decided to use JavaScript, because of its simplicity, and p5.js for its easy setup and visualizations. When creating the project, I decided to add a bit more physics to the algorithm, therefore I applied mass to each boid, and having the steering of that boid a bit harder due to its mass. This made the smaller boids more agile, while the larger boids very bulky.

The images below are code snippets showcasing the behavior of each rule (mentioned above) that every boid follows.

I am aware of the time complexity of this project, and there are many more efficient ways of improving it. One way I could think of is to put the boids in a tree so we can iterate through them faster if there are close to the main boid

Minimum Spanning Tree + A*

  • Project Status: Complete
  • Project Type: Personal
  • Project Duration: ~1 days
  • Software Used: VS Code
  • Languages / Libraries Used: JavaScript, p5.js

After reading the "Introduction to algorithms", by Thomas H. Cormen, Charles E. Leiserson, Ronald L. Rivest, and Clifford Stein. I wanted to expand my knowledge regarding algorithms. I wanted to recreate the minimum spanning tree in a more interactable way, and then allow the user to choose a starting and ending point, and apply a path finding algorithm to the tree.

I have only implemented the A* algorithm (although not really worth it for this project), which I had a lot of troubles with. I learned a lot by trying to implemented my theoretical knowledge to practice, and I recommend anyone else who wants to learn algorithms to try and implement the algorithms that they find interesting.

You can see a snippet of the implementation for the A* algorithm below, and more precisely the formula.

I had to apply a heuristics function to the algorithm, and I chose the raw euclidean distance instead of the manhattan distance for example.

Start by clicking on the canvas to create tree nodes. These nodes will be connected between each other with edges based on the Minimum Spanning Tree algorithm. After you have placed 20 nodes, you can click on any node to make it a START node. Then click on another node to make it an END node. Finally see the shortest path based on the A* algorithm.

Visualizing A* with SFML

  • Project Status: Complete
  • Project Type: Personal
  • Project Duration: ~3 days
  • Software Used: Codeblocks
  • Languages / Libraries Used: C++, SFML

I began my journey into algorithms as any software engineer would. Through watching youtube videos. I was amazed by the sheer number of cool things people could make using algorithms. A few weeks before this project I picked up the "Introduction to algorithms", by Thomas H. Cormen, Charles E. Leiserson, Ronald L. Rivest, and Clifford Stein, and the "Algorithm Design Manual" by Skiena. There were so many algorithms that I knew from my Bachelors education, but on the other hand there were also so many that I had never heard about. I dived deeper and deeper into pathfinding algorithms as that piqued my interest.

To implement the A* algorithm I decided that I needed a language that is very efficient when dealing with computational power, therefore I picked c++. I also needed a way to visualize everything since its pretty boring seeing your console line outputting this:

Starting Node = {0, 0}
Target Node = {99, 99}
Algorithm Starting...
Computing shortest path...
Shortest path found!

For the graphics I chose to use the SFML library as it is easy to pick up, and I was satisfied with having 2d graphics, as I only needed to represent a 2D grid.

The hardest part was fiquring out how to implement the algorithm in c++, since I am new to the language and there are many simple things I did not know. For example in Java I could use a List for the Open and Closed sets, and since the list has methods already implemented for finding if an element is in the list (Contains) and remove, the algorithm would be way easier to implement.

Another obstacle I ran into was optimization. By using a normal array (vector) in c++, finding the node with the lowest f cost was very time consuming. I had to implement a min heap to avoid this overhead and that's when other problems occured. I am relatively new to using pointers, and using them efficiently is another task. In the end through trial and error I managed to make the algorithm work by replacing the Open and Closed set with a min heap.

Code before and after min heap implementation

I decided to track how much time it would take to run each algorithm (with min heap and without) by using a timer. I implemented a Timer class by following this tutorial.

Maze Generator

  • Project Status: Complete
  • Project Type: Personal
  • Project Duration: ~2 days
  • Software Used: Codeblocks
  • Languages / Libraries Used: C++, SFML

For this project I used a maze generating algorithm to draw a NxN maze. At first I thought it would be a simple task, but I encountered a couple of blockers that made me rethink the complexity of the algorithm.

The first obstacle that I encountered was random. It was literally the random number generator in c++. I noticed that rand() is not as random as one might think. Without knowing this information I thought that my algorithm wasn't working properly, but after thorough research I found new ways to make a random number generator in c++.

Choosing a random neighbor

To draw the walls of the cell I decided to draw 4 individual rectangles to represent each wall (up, down, left, right). This made it easier when making the made, since in order to connect two cells together, I would just destroy the walls that are inbetween them.

The four walls of each cell

Implementing the stack was relatively easy, since I know the data structure and have used it previously. The only confusing thing was that when calling pop(), it did not return the actual element from the stack. I had to use top() to get the element, and then call pop().

The hardest part by far was the actual visualization. I had to find a way to visualize each cell whenever it was visited, or backtracked from the stack.

Popping the top most cell from the stack

DirectX 11 Sandbox

  • Project Status: In Progress
  • Project Type: Personal
  • Project Duration: October 3rd, 2020 - Present
  • Software Used: Visual Studio 2019
  • Languages / Libraries Used: C++, DirectX11, HLSL, win32, ImGui

This is a project I wanted to tackle for a very long time. I have read and heard hundreds if not thousands of blog posts, forums, opinions, and books saying to never try and create a game engine on your own. I know it is a task that one can fully achieve only in a lifetime (still debatable), but I wanted to do it anyways. I don't want to recreate the next Unity or Unreal, but a small sandbox for me to play around with and learn new things about the DirectX API.

After reading Jason Gregory's 'Game Engine Architecture' book, I really wanted to go through that process myself. DISCLAIMER! I do not want to make game engines or renderers. I want to be a game developer, but what good is it making games if I can't understand the technology behind them?

Having said all of this, I began my project by researching. As always I watched through countless of videos, read two more books ('3D game programming with directX 11' by Frank D. Luna, and '3D Game Engine Architecture' by David H. Eberly) until I reached a point where I felt somewhat confident in tackling it on my own (Spoiler alert, I had plenty of external help).

I know some people can judge and say "Hey you copied this from *insert fictional palce here*!!!", and yes I did, but frankly I do not care. This is a learning experience and I don't think any human can tackle game engines completely alone from the first try.

I began by following some video tutorials by ChilliTomatoNoodle on youtube. He explained a lot of things that were unclear to me, and taught me a lot of useful techniques not only for DirectX 11, but C++ as well.

Still learning and trying to find any free time I can to work on this project, I am extremely happy I have finally decided to do something about my dream.
